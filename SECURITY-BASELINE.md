# SECURITY BASELINE — 全プロジェクト共通のセキュア・バイ・デフォルト規約

> このファイルは Generator の実装指針であり、Security-auditor の監査チェックリストです。
> **Mock / POC であっても顧客に見せる前提**で、ここに反する実装は不合格です。

## 1. 秘密情報 (Secrets)
- API キー・トークン・パスワード・接続文字列を**ソースコード/リポジトリに書かない**
- 秘密は環境変数（`.env`）から読む。リポジトリには `.env.example`（プレースホルダのみ）だけをコミット
- `.env`, `*.pem`, `id_rsa`, 認証情報ファイルは `.gitignore` 済みであることを確認
- フロントエンドのバンドルに秘密を埋め込まない（公開鍵/公開可能値のみ。`NEXT_PUBLIC_` 等の扱いに注意）
- ログ・エラーメッセージ・URL クエリに秘密やトークンを出さない

## 2. 入力検証 (Input Validation)
- すべての外部入力（フォーム、クエリ、ヘッダ、Cookie、ファイル、Webhook）は**サーバ側で検証**する
- 許可リスト方式（型・長さ・範囲・形式・列挙）。クライアント検証は UX 用であり信頼しない
- スキーマバリデーション（zod / pydantic 等）を入口に置く

## 3. インジェクション対策
- SQL/NoSQL: パラメータ化クエリ / ORM のバインド。文字列連結でクエリを作らない
- OS コマンド: 原則使わない。必要なら引数配列で渡し shell=false。ユーザー入力を渡さない
- パストラバーサル: ファイルパスにユーザー入力を使う場合は正規化＋ベースディレクトリ拘束
- テンプレート/eval: ユーザー入力を `eval`/動的テンプレートに渡さない

## 4. 出力エスケープ / XSS
- デフォルトでエスケープされる仕組みを使う（React/Vue のテキストバインド等）
- `dangerouslySetInnerHTML` / `v-html` / `innerHTML` に未サニタイズのユーザー入力を渡さない
  - HTML を出す必要があれば DOMPurify 等でサニタイズ
- ユーザー生成リンクは `javascript:` 等の危険スキームを禁止

## 5. 認証・認可 (AuthN / AuthZ)
- 認証が必要な操作・データは**サーバ側で毎回認可を検証**する
- IDOR 防止: リソース取得/更新時に「そのリソースが現在のユーザーのものか」を必ず照合
- 認可をフロントの表示制御だけに頼らない（API 単体で防御）
- パスワードは bcrypt/argon2 等でハッシュ。平文・可逆暗号で保存しない
- セッション/JWT: 適切な有効期限、サーバ側失効手段、署名検証

## 6. セッション / Cookie / CSRF
- 認証 Cookie は `HttpOnly` + `Secure` + `SameSite=Lax`（または Strict）
- 状態変更操作（POST/PUT/DELETE）は CSRF 対策（SameSite + トークン or ダブルサブミット）
- ログイン後・権限変更時にセッション ID を再生成

## 7. SSRF / 外部リクエスト
- ユーザー入力の URL へサーバからリクエストする場合、許可ドメインのリストで制限
- クラウドメタデータ（169.254.169.254 等）や内部 IP レンジへの到達を遮断
- リダイレクト追跡時もホスト検証を維持

## 8. 依存関係 (Supply Chain)
- lockfile（package-lock.json / pnpm-lock.yaml / poetry.lock 等）を必ずコミットしてバージョン固定
- 既知脆弱性のあるバージョンを使わない（`npm audit` / `pip-audit` 相当の観点でチェック）
- 不要な依存を入れない。出所不明なパッケージを避ける

## 9. ファイルアップロード（該当時）
- 拡張子だけでなく MIME / マジックバイトを検証、サイズ上限、保存名はサーバ生成
- アップロード先は実行権限のない領域。アップロードファイルをそのまま配信しない（必要なら別オリジン）

## 10. 設定 / ヘッダ / エラー処理
- 本番ビルドでデバッグ/詳細スタックトレースをユーザーに出さない（汎用エラーメッセージ）
- CORS: `Access-Control-Allow-Origin: *` と credentials を併用しない。許可オリジンを明示
- セキュリティヘッダを付与: `Content-Security-Policy`（可能な範囲）, `X-Content-Type-Options: nosniff`, `Referrer-Policy`, `X-Frame-Options`/frame-ancestors
- open redirect 防止: リダイレクト先は許可リスト or 相対パスのみ
- レート制限（少なくとも認証・書き込み系エンドポイント）

## 11. ログ / 監査
- 認証失敗・認可拒否・重要操作をログ（ただし秘密・個人情報はマスク）
- 例外を握りつぶさない。失敗時は安全側（deny）に倒す

## 12. 画像・メディアのライセンス遵守

- 第三者の画像を使う場合は **必ず再配布可能なライセンス**を確認する。`docs/images.json` と `docs/image-credits.md` で管理
- 採用可: **CC BY / CC BY-SA / CC0 / Public Domain**
- 採用不可: **NC（非商用限定）/ ND（改変禁止）**、ライセンス不明
- 著者名（Artist）と出典URL（Wikimedia Commons の File ページなど）を必ず記録
- 画像は **ローカルに保存**して配信。外部CDNからの直接 hot-link をしない（規約・速度・firewall 整合性）
- UI 上で各画像のクレジット（著者・ライセンス）を視認可能な形で表示
- Unsplash 等のサービス経由で取得した画像は、各サービスの規約（写真家プロフィールへのクリック可リンク等）を遵守

## 13. サンドボックス運用ルール
- 外部送信はコンテナのファイアウォールで allowlist 制限されている
- 許可外ドメインへの通信が必要になっても、ファイアウォールを勝手に緩めない。ユーザーに報告し判断を仰ぐ
- ホストの秘密情報（~/.ssh, ~/.aws 等）はコンテナに存在しない前提。要求しない

---
### 監査の合否ルール（Security-auditor 用）
- Critical / High が 1 件でもあれば **不合格**（@generator へ差し戻し）
- Medium は原則不合格（修正必須）。軽微な Low のみ次スプリント改善で合格維持可
- 「該当機能が未実装だからチェック不要」な項目は *N/A* と明記（黙ってスキップ禁止）
